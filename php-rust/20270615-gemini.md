# Ricognizione Progetto `php-rust` (PHPR) - Analisi e Prospettive

Questo documento fornisce una ricognizione dettagliata dello stato attuale del progetto `php-rust` (re-implementazione di PHP 8.5.7 in Rust), identificando il livello di maturità, i margini di miglioramento architetturali e prestazionali, le criticità lungo la timeline e le potenzialità per rendere il prodotto uno standard assoluto di settore ("best in class").

---

## 1. Stato di Maturità del Progetto

Il progetto ha raggiunto un livello di maturità **eccellente e sorprendente**. Non si tratta di un banale subset "giocattolo" del linguaggio, ma di una Virtual Machine in grado di eseguire codice PHP reale garantendo la **byte-compatibility** (bug-for-bug compatibility) con lo Zend Engine originale.

- **Successi raggiunti:** La capacità di far girare Composer, PHPUnit (incluso process isolation), Doctrine ORM/DBAL, componenti di Symfony (es. http-kernel al 100% senza fallimenti) e soprattutto **WordPress** (wp-cli ed engine core frontend/backend con SQLite) dimostra una solidità del Core Engine (AST, Compiler, VM) straordinaria.
- **Supporto linguistico avanzato:** Il supporto di feature complesse del linguaggio moderno (Fibers, Generators, Attributi, Enums, Type Juggling, destructuring `list()`, ecc.) e di una vera Object-Oriented Programming (OOP) è già integrato e funzionante.
- **Metriche:** L'80% delle funzioni della libreria standard (*core stdlib*) è completato, con oltre il 61% del corpo dei test ufficiali Zend (Zend test corpus) superato con successo.

## 2. Suggerimenti per il Miglioramento (Performance, Chiarezza, Modularità)

Sebbene il motore funzioni incredibilmente bene, il codice presenta forti debiti tecnici dovuti alla rapida iterazione, che vanno sanati per preparare il terreno ai contributor e alle ottimizzazioni avanzate.

### A. Modularità e Chiarezza (Refactoring Urgente)
L'architettura soffre attualmente di un'elevata **monoliticità**, un classico collo di bottiglia per manutenibilità e risoluzione di conflitti (git conflicts):
- **`vm/mod.rs` (oltre 24.000 righe):** È il problema principale. Contiene più di 200 implementazioni di *host builtins* (funzioni prefissate con `ho_`). **Soluzione:** Tutti i metodi `ho_*` devono essere estratti e spostati verso il crate `php-builtins` oppure raggruppati in sottomoduli dedicati (es. `vm/host_funcs/`).
- **`compile.rs` (oltre 5.300 righe) e `lower/mod.rs` (oltre 6.500 righe):** Entrambi necessitano di essere "spaccati" in directory modulari (`compile/expr.rs`, `compile/stmt.rs`, `compile/class.rs`), replicando il pattern di parziale successo già iniziato in `lower`.
- **Crate `php-builtins`:** I file `file.rs`, `string.rs` e in particolare `lib.rs` (che non dovrebbe contenere logica applicativa ma solo definizioni e tipi) stanno crescendo fuori controllo e richiedono una suddivisione logica (es. estrarre `var_dump` e controlli sui tipi in file separati).

### B. Performance
L'introduzione dell'allocatore globale `mimalloc` e di `FxHash` (rustc-hash) durante le ultime sessioni WP ha generato un eccellente **-35% nei tempi di risposta**. Per andare oltre, serve affrontare l'architettura della memoria:
- **Riduzione del "Churn" (Clone/Drop):** Attualmente il collo di bottiglia risiede nei frequenti `clone` e `drop` delle strutture dati `Zval`.
- **Arena Per-Request:** Implementare un sistema di memoria basato su *arena* per le singole richieste, simile allo Zend Memory Manager (ZMM). Al termine della richiesta HTTP, l'intera arena viene deallocata istantaneamente a costo (quasi) zero, annullando il peso computazionale di spazzare via il garbage.
- **Interning dei Simboli:** Le stringhe usate come chiavi, nomi di funzioni o classi, dovrebbero essere internate (convertite in interi come `u32`). Confrontare interi (O(1)) è infinitamente più veloce rispetto alla comparazione di stringhe per la risoluzione dei metodi e l'accesso agli array associativi.

## 3. Criticità e Ostacoli (Breve, Medio e Lungo Periodo)

### Breve Periodo (Attuali)
- **Debito di Manutenibilità:** Se non si spacchettano i file giganti sopra menzionati, ogni nuova estensione aumenterà la frizione nello sviluppo, causando conflitti di merge insormontabili quando ci saranno più contributor.
- **Feature Gap per WP:** L'ecosistema reale (es. WordPress) necessita di estensioni complete come `mysqli` e `gd`. L'integrazione con un client MySQL nativo Rust è il passo più imminente da superare per rendere la VM utile su stack comuni.

### Medio Periodo
- **Gestione dei Riferimenti Circolari (Garbage Collection):** Attualmente la gestione della memoria sfrutta `Rc<RefCell<T>>`. Poiché PHP tollera e crea costantemente riferimenti circolari, `Rc` non è in grado di liberarli (generando Memory Leak). Scrivere un **Cycle Collector** o un Tracing Garbage Collector in Rust, che sia al tempo stesso safe e iper-performante, è una delle sfide di ingegneria informatica più grandi di questo progetto.

### Lungo Periodo
- **L'Ecosistema PECL (C Extensions):** Zend Engine deve il suo successo alle migliaia di estensioni scritte in C (Xdebug, Redis, Imagick). Per supportarle senza doverle riscrivere, sarà necessario creare un layer FFI (Foreign Function Interface) che permetta alle librerie C di interfacciarsi con le `Zval` di Rust. Un lavoro di astrazione gigantesco e complesso.
- **Bug-for-Bug Parity (The Long Tail):** Gli ultimi 20% dei comportamenti di PHP (warning legacy, silenti conversioni di tipo, bizzarrie storiche) richiederanno un dispendio di energie sproporzionato per mantenere intatta la compatibilità con i framework più esigenti.

## 4. Direzione per diventare "Top Best in Class"

Per trasformare `phpr` da un "semplice" porting di successo a un ecosistema che supera PHP stesso, le direttrici chiave sono tre:

1. **Async PHP Nativò (Il Deno per PHP):** 
   Visto che i Fiber sono già parcheggiati sull'heap (allocazione dinamica), la VM è semanticamente pronta per il multi-threading vero e proprio. Integrando un event-loop come **Tokio**, `phpr` permetterà di scrivere codice PHP che scala orizzontalmente in modo asincrono, reggendo decine di migliaia di WebSocket senza necessitare di sovrastrutture come Swoole o RoadRunner. È una "killer feature" che cambierebbe il paradigma del linguaggio.

2. **Distribuzione "Single-Binary" (Il modello Go):**
   Incorporare un server HTTP (come `hyper` in Rust) direttamente all'interno della VM, distribuendo `phpr` come un unico file eseguibile standalone. Questo eliminerà il fardello devops di configurare PHP-FPM, Apache/Nginx e decine di librerie `.so`, permettendo deploy immediati (stile container-less).

3. **JIT Compilation (Tier 3):**
   Avendo un bytecode fortemente tipizzato e molto più pulito di Zend, `phpr` ha l'opportunità di introdurre un compilatore Just-In-Time (es. tramite *Cranelift* o *LLVM*) moderno. Il codice sorgente verrebbe trasformato direttamente in codice macchina in fase di esecuzione, spazzando via le performance di Zend Engine.

4. **Community e Open Source:**
   Rust ha una community incredibilmente attiva. Sfruttare la "magia" della dicitura "PHP in Rust" lanciando ufficialmente il progetto al grande pubblico (dopo aver stabilizzato Composer e il routing base di Laravel/WP) attrarrà inevitabilmente centinaia di dev pronti a sottomettere Pull Request per implementare gli opcode e le funzioni della standard library mancanti, spalmando lo sforzo su larga scala.
