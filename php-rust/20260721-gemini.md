## 1. BERSAGLIO PRINCIPALE: Rimozione dell'`op-clone` in `run_loop`
Claude ha confermato: la clonazione dell'`Op` alla riga 92 di `run.rs` è la prossima "grande leva" per abbattere l'overhead del `run_loop`.

**Il meccanismo corretto (Zero-Cost Borrow):**
Poiché `Frame.func` è in realtà un `&'m Func` (una reference `Copy` con il lifetime del modulo), non c'è alcun bisogno di `Rc::clone`. È sufficiente copiare la reference fuori dalla catena di accessi per liberare `self`, e fare il match per reference:
```rust
let func = self.frames[top].func; 
let op = &func.ops[ip]; // match per reference!
```
**Il Refactor:**
Questo richiederà di toccare il cuore della VM. Le centinaia di *arm* del `match` attualmente legano i payload per valore (es. `Op::LoadVar { name } => ...`). Dovranno essere aggiornati per legarli per reference (es. `Op::LoadVar { name } => ...` dove `name` diventerà `&ConstIdx` o la reference corretta). 
*Nota rassicurante:* Le Inline Caches (MethodIc, PropIc) basate su `Rc`-condivise introdotte nelle ultime sessioni non si romperanno, manterranno la loro correttezza.

Questo task è grande ma essenziale: smetteremo di allocare e muovere memoria inutilmente a ogni singolo tick dell'orologio della VM.

## 2. Micro-ottimizzazione Immediata: Pre-computing in `bind_params`
Riguardo al costo di `enter_callee/bind_params`, una specializzazione JIT è speculativa, ma Claude ha individuato un'ottima "low-hanging fruit".

**Azione:**
Invece di eseguire `param_hints.iter().any(...)` a ogni singola chiamata di funzione, calcolare un flag `has_hints: bool` sulla `Func` a *compile-time*. Questo salverà cicli preziosi in uno dei percorsi più caldi e inevitabili del binding degli argomenti.

## 3. Roadmap a Medio Termine (Confermata)
Gli altri colli di bottiglia identificati dal profilo andranno gestiti nelle roadmap future:
- **Value Churn / `Zval` footprint:** Avendo già confermato che `Zval` è fisso a 16 byte, la strada futura è il **NaN-boxing** (impacchettamento del tipo e del payload nei 64-bit IEEE 754), che è la soluzione definitiva al value-churn usata da V8 e dallo stesso Zend Engine moderno.
- **String Interning (Symbol u32):** Molto sensato per pulire i confronti di stringhe statiche, ma il ROI è mitigato dal fatto che gli hash sono già cachati (`zhash`) e le chiavi degli array dinamici runtime continuerebbero comunque a eludere l'interning statico.
